# Екзаменаційний білет №17: Принцип єдиної відповідальності (SRP)

Цей проект присвячений практичній демонстрації одного з ключових принципів **SOLID** — **Single Responsibility Principle** (Принцип єдиної відповідальності) засобами мови TypeScript.

## Мета завдання
Створити гнучку та масштабовану архітектуру, продемонструвавши перехід від "антипатерну" (брудного коду) до "кращих практик" (рефакторингу).

## Що було зроблено:

### 1. Реалізація антипатерну (`src/SRP/antipattern/`)
Створено клас `OrderProcessor`, який є прикладом "Божественного об'єкта" (God Object). Він порушує SRP, оскільки поєднує в собі три різні відповідальності:
* Обробка бізнес-логіки замовлення.
* Безпосередня робота з базою даних.
* Логіка відправки сповіщень користувачу.
*Порушення полягає в тому, що будь-яка зміна (наприклад, зміна формату БД або сервісу розсилки) змушує нас правити цей єдиний клас.*

### 2. Рефакторинг та розділення відповідальностей (`src/SRP/refactored/`)
Код було розділено на три незалежні модулі, кожен з яких має лише одну причину для зміни:
* **`OrderRepository`**: відповідає виключно за збереження та отримання даних.
* **`NotificationService`**: відповідає лише за відправку повідомлень (Email/SMS).
* **`OrderService`**: виконує роль координатора, не вникаючи в деталі реалізації БД чи пошти.

## Для чого це потрібно?
Такий підхід забезпечує:
1.  **Тестованість**: ми можемо окремо протестувати збереження в базу, не відправляючи реальних листів.
2.  **Масштабованість**: якщо ми вирішимо замінити SQL базу на NoSQL, нам потрібно буде змінити код лише в одному класі (`OrderRepository`).
3.  **Читабельність**: код став простішим, структурованим та легшим для розуміння іншими розробниками.
